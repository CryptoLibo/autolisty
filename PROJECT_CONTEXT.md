# Autolisty Project Context

Autolisty is a private production app used to prepare and publish digital Etsy listings, plus supporting Pinterest content. It is not a public SaaS product and is operated with one Etsy account and one Pinterest account.

This document exists so future Codex sessions can understand the app before making changes. Do not add secrets to this file.

## Operating Model

- Local machine is mainly used for code edits with Codex.
- Production is the real validation environment.
- Normal update flow:
  1. Make or review changes locally.
  2. Push changes from the local app console with Git.
  3. SSH into the Hetzner server from a separate console.
  4. Run `./deploy.sh` on the server.
- `./deploy.sh` lives on the Hetzner server, pulls from GitHub, builds/deploys the app, and restarts it with PM2.
- GitHub is the source of truth for code.
- The server and GitHub are kept aligned by pushing locally first, then deploying from the server.

## Security Rules

## Domains

- Main app: `autolisty.com`
- Delivery download subdomain: `download.autolisty.com`

`download.autolisty.com` is handled by a Cloudflare Worker. Delivery PDFs generated by the app contain invisible link areas/buttons that point to this subdomain. When a customer clicks a PDF button, the Worker resolves the correct final asset and triggers the download.

Do not change the hardcoded `download.autolisty.com` link patterns unless the Cloudflare Worker and R2 object layout are being changed together.

## Integrations

### OpenAI

Used for:

- Etsy SEO generation.
- Etsy title, description keywords, tags, and mockup alt text.
- Pinterest title and description generation.
- Prompt Lab image analysis and prompt creation.

OpenAI routes currently use `gpt-4o` in code.

### Etsy

Used for:

- OAuth connection.
- Reading the connected shop.
- Updating an existing Etsy draft listing.
- Replacing listing images.
- Uploading image alt text when Etsy accepts it.
- Uploading the final delivery PDF as the Etsy digital file.
- Fetching a public listing URL after the listing is published.

Important behavior:

- The app expects the user to provide an existing Etsy draft listing ID.
- The listing ID must be numeric for Etsy sync.
- The app uses the first Etsy shop returned by the account.
- The current private workflow assumes one Etsy account and one shop.

### Pinterest

Used for:

- OAuth connection.
- Reading boards.
- Generating Pinterest copy.
- Publishing organic pins to the selected board.

The current private workflow assumes one Pinterest account.

### Cloudflare R2

Used for storing:

- Design images.
- Etsy mockups.
- Listing videos.
- Deliverable source files.
- Generated delivery PDFs.
- Pinterest images.

Current object layout:

- `mockups/{listingId}/...`
- `deliverables/{listingId}/...`
- `pinterest/{listingId}/...`
- `delivery-pdf/{listingId}.pdf`

The app includes cleanup logic for generated listing assets.

## Folder Import Structure

The app recognizes a complete listing folder like this:

```text
Listing folder/
  Diseños Finales/
    final deliverable files
  Pines/
    Pinterest images
  1.jpg
  2.mp4
  3.jpg
  4.jpg
  5.jpg
  6.jpg
  7.jpg
  8.png
  9.jpg
  10.jpg
  11.jpg
  Midjourney.png
  Upscayl.png
```

Recognition rules:

- A filename containing `Midjourney` is treated as the main design image.
- A filename containing `Upscayl` is ignored.
- Numbered root images such as `1.jpg`, `3.jpg`, `4.jpg` are treated as Etsy listing mockups and sorted numerically.
- Root video files such as `2.mp4` are treated as the optional Etsy listing video.
- `Pines` or `Pins` contains Pinterest images.
- `Diseños Finales` contains the final customer deliverable assets.

Frame TV Art deliverables:

- One final artwork image.
- Optional instructions PDF.

Printable Wall Art deliverables:

- Image files named so the app can identify ratios:
  - `2x3` or `2-3`
  - `3x4` or `3-4`
  - `4x5` or `4-5`
  - `11x14` or `11-14`
  - `ISO`

Do not change these recognition rules casually because they are part of the working production workflow.

## Delivery PDF Flow

The app generates a delivery PDF from a template in `templates`.

Current templates:

- `templates/frame_tv_delivery.pdf`
- `templates/print_art_delivery.pdf`

Generation flow:

1. User uploads deliverable files.
2. Files are stored in R2 under `deliverables/{listingId}/`.
3. App creates a PDF from the product template.
4. App injects invisible link annotations pointing to `download.autolisty.com`.
5. Generated PDF is uploaded to R2 under `delivery-pdf/{listingId}.pdf`.
6. Etsy sync uploads this generated PDF as the Etsy digital file.

The Cloudflare Worker behind `download.autolisty.com` is responsible for turning those links into direct downloads.

## Development Notes

- Main scripts:
  - `npm run dev`

  - `npm run lint`

- Before changing Etsy, Pinterest, R2, PDF, or folder import behavior, read the related files end to end.

## Safe Change Guidelines

For future Codex work:

- Start by checking `git status --short --branch`.
- Do not modify `.env.local`.
- Do not expose secret values.
- Keep changes small and targeted.
- Preserve the current folder import structure.
- Preserve R2 key structure unless the Cloudflare Worker is updated too.
- Preserve `download.autolisty.com` PDF links unless intentionally changing the download architecture.
- Preserve the private single-account assumption unless the user asks for multi-account support.
- If Node/npm are available locally, run `npm run lint` or `npm run build` when relevant.
- If local validation is not possible, say so clearly and keep the change easy to review before push/deploy.
